# Gemini 3.1 Pro Preview: MALFORMED_FUNCTION_CALL Failures in Eval Harness

**Date:** 2026-03-06
**Severity:** High -- blocks Gemini 3.1 evaluation runs
**Affected Model:** `gemini-3.1-pro-preview` (via Google AI Studio / `google-genai` SDK)
**Status:** Open

---

## Executive Summary

During puzzle evaluation runs on 2026-03-06, the Gemini 3.1 Pro Preview model exhibited persistent `MALFORMED_FUNCTION_CALL` errors, returning HTTP 200 responses with empty content parts. This caused repeated retries, wasted API spend, and severely degraded run throughput. All other evaluated models (GPT 5.2, Claude Opus 4.6, Kimi k2.5) completed their runs without this issue.

---

## Impact

- **Run throughput:** Each malformed response triggers 3 provider-level retries + runner-level backoff (up to 5 consecutive skips before abort). A single bad step burns ~3 minutes of wall time and 3x the token cost with no progress.
- **Cost waste:** Every retry re-sends the full prompt (including base64 image). At Gemini 3.1 pricing, each wasted call costs approximately $0.01-0.02 in input tokens alone.
- **Run completion risk:** After 5 consecutive skips the runner aborts the run entirely, producing incomplete evaluation data.
- **Comparison validity:** Incomplete or aborted Gemini runs cannot be fairly compared against other models that completed all runs cleanly.

---

```
21:48:27 | INFO    | httpx | HTTP Request: POST ... "HTTP/1.1 200 OK"
21:48:27 | WARNING | eval_runner | Provider returned SKIP (skip 1/5): (empty response: content.parts is None/empty (finish_reason=FinishReason.MALFORMED_FUNCTION_CALL))

```

This pattern repeated multiple times, with the model consistently returning `finish_reason=FinishReason.MALFORMED_FUNCTION_CALL` across all 3 internal retries before escalating to the runner's skip counter.

### Key Observations

1. **HTTP status is 200 OK** -- the API does not report an error. The response body contains a candidate with `finish_reason=MALFORMED_FUNCTION_CALL` and no usable content parts.
2. **Retries do not help** -- the same prompt retried seconds later produces the same malformed result, suggesting a deterministic model-level failure rather than a transient infrastructure issue.
3. **The failure is intermittent across steps** -- the model successfully returned function calls for earlier steps (e.g., Step 0 action=UP at 21:43:39), then started failing consistently at later steps, possibly as conversation context grew.

---

## Root Cause Analysis

### All 4 Providers Use Forced Tool Calling

Every provider in the harness uses the same pattern: define a `play_action` tool/function and force the model to call it. The API syntax differs per vendor, but the semantics are identical:

| Provider | Force Mechanism | Result |
|---|---|---|
| OpenAI (GPT 5.2) | `tool_choice: {type: "function", function: {name: "play_action"}}` | Works |
| Anthropic (Claude Opus 4.6) | `tool_choice: {type: "tool", name: "play_action"}` | Works |
| Bedrock Claude | `toolChoice: {tool: {name: "play_action"}}` | Works |
| Bedrock Kimi (k2.5) | `toolChoice: {tool: {name: "play_action"}}` | Works |
| **Gemini 3.1 Preview** | `function_calling_config(mode="ANY", allowed=["play_action"])` | **Fails** |

### The Problem Is Model-Side

- `MALFORMED_FUNCTION_CALL` is a Gemini-specific `FinishReason` indicating the model attempted to generate a function call but produced structurally invalid output internally.
- This happens *before* our code parses anything -- the API itself reports the malformation.
- The tool schema is simple and correct (3 string fields, 2 required). Other models handle an equivalent schema without issue.
- The model is a **preview** release (`gemini-3.1-pro-preview`), which is expected to have rough edges.

### Ruling Out Prompt Conflict

Initial hypothesis was that the shared prompt's `"Respond with JSON: {...}"` instruction conflicted with function calling. However, **all 4 providers receive the same prompts with the same JSON instructions**, and only Gemini fails. The prompt conflict may exacerbate the issue but is not the root cause.

## Recommended Actions

### Short-Term Fix (Recommended)

**Drop function calling for Gemini; use text-mode JSON parsing instead.**

- Remove the `tools` and `tool_config` parameters from the Gemini API call.
- Let the model respond with plain text JSON (which the prompt already requests).
- Parse the response using the existing `BaseProvider._parse_action_response()` method, which all providers already have as a fallback.
- This requires changes to **one file only**: `scripts/evaluate/providers/gemini_provider.py`.
- Risk: slightly less structured output, but the text parser has robust fallback extraction (JSON regex, keyword scan, first-valid-action default).

### Medium-Term

- **Monitor for GA release:** When `gemini-3.1-pro` (non-preview) ships, re-test function calling. Preview model bugs often get fixed at GA.
- **Add a provider capability flag:** Add `supports_function_calling: bool` to `ModelConfig` so the harness can automatically choose text-mode for models known to be unreliable with tool use.

### Not Recommended

- **Stripping JSON instructions from prompts** -- all other providers work fine with the current prompts. Changing shared prompts to fix one model's bug adds complexity and risk.
- **Waiting for Google to fix it** -- preview models have no SLA for bug fixes, and we need evaluation data now.
